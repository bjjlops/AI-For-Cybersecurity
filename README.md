# Final Project: Build an AI Pentesting Agent

Your task is to build an autonomous AI agent that conducts penetration testing against [OWASP Juice Shop](https://owasp.org/www-project-juice-shop/), a deliberately vulnerable web application.

This project evaluates your ability to **design** an agent — not your ability to prompt one. The scaffold provides the plumbing (LLM abstraction, HTTP client, scoring harness). You provide the brains (planner, prompts, attack strategy, specialized tools).

## Learning objectives

By completing this project you will:
1. Design and implement an autonomous agent loop with tool use
2. Translate pentesting methodology (recon → vuln discovery → exploitation → reporting) into machine-executable logic
3. Evaluate how prompt engineering and tool design affect agent performance on security tasks
4. Experience the practical limits of LLM-driven automation on adversarial tasks

---
## Prerequisites
1. You must have Docker installed on your computer.  (https://www.docker.com)
2. You must have an API key for your LLM of choice.  The instructions below explain how to configure the scaffolding to use whatever LLM you want.
3. Python with virtual environments.

## Setup

### 1. Run Juice Shop

```bash
docker compose up -d juiceshop
```
Verify it is running at http://localhost:3000.

### 2. Install dependencies

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 3. Configure your LLM

Copy `.env.example` to `.env` and set the credentials for whichever provider you want to use. The scaffold uses [LiteLLM](https://docs.litellm.ai/), so you can use OpenAI, Anthropic, Google, Azure, Ollama (local), Groq, Together, or any of 100+ providers by changing one env var.

Be sure to uncomment the line beginning with `LLM_MODEL` and paste your API key in the appropriate variable.

Examples:
```bash
# OpenAI
LLM_MODEL=gpt-4o
OPENAI_API_KEY=sk-...

# Anthropic
LLM_MODEL=anthropic/claude-sonnet-4-5
ANTHROPIC_API_KEY=sk-ant-...

# Local via Ollama
LLM_MODEL=ollama/llama3.1:70b
OLLAMA_API_BASE=http://localhost:11434
```

### 4. Sanity check

```bash
python -m eval.harness --dry-run
```

This verifies Juice Shop is reachable and your LLM credentials work.

**DO NOT PROCEED UNLESS THIS STEP COMPLETES SUCCESSFULLY**

## What you need to build

The scaffold has deliberate gaps. Your job is to fill them.

### Required deliverables

1. **`agent/agent.py`** — The agent loop. The stub defines the interface; you decide the architecture (ReAct, Plan-and-Execute, multi-agent, hierarchical — your choice, justify it in the design doc).
2. **`agent/prompts/`** — System prompt(s) and any task-decomposition prompts. Empty templates are provided.
3. **At least 3 new tools** in `agent/tools/` beyond the two reference tools. Tools must be substantive — a "wrapper that calls another tool" does not count. Examples of acceptable tools: SQL injection tester, JWT decoder/forger, XSS payload generator, directory brute-forcer, API fuzzer, business-logic probe.
4. **Final pentest report** (`reports/final_report.md`) — Generated by your agent at the end of a run. Must include findings, severity, evidence, remediation.
5. **Design document** (`DESIGN.md`) — See `DESIGN_DOC_TEMPLATE.md`. This is where you justify every design decision. **This is graded heavily.**

### What you may NOT change

- `eval/score.py` and `eval/harness.py` — grading uses these as-is
- `agent/llm.py` — the LLM abstraction is fixed so everyone competes on equal LLM footing
- `agent/tools/base.py` — the tool protocol

Everything else is yours.

## Running your agent

```bash
python -m agent.main --max-steps 100 --target http://localhost:3000
```

Scoring:

```bash
python -m eval.harness --report
```

This queries Juice Shop's `/api/Challenges` endpoint and reports which challenges your agent solved.

## Academic Integrity and AI Usage

You MAY use LLMs as coding assistants. You may NOT submit LLM-generated code you do not understand. The design document and writeup exist to verify this — they ask for failure analysis, justified architectural choices, and annotated screenshots of your agent running. LLM-generated submissions consistently fail on these sections because they cannot fabricate authentic debugging narratives or screenshots.

Cite any code you adapt from external sources (blog posts, GitHub, AI assistants) in `DESIGN.md`.
